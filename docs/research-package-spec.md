# CollabVaultX Research Package Spec

This document defines the input package that GPT Pro should produce before
Codex imports or generates any campaign pages.

The package is the contract between research and generation. GPT Pro researches
one collaboration campaign and outputs a single JSON package with sources,
evidence, assets, item groups, marketplace search metadata, and page-generation
decisions. Codex validates that package, imports it into the site data model,
and generates the campaign page plus selected item pages.

## Goals

- Stop manually adding individual item pages as the main development path.
- Keep the research source of truth outside `data/` until it passes validation.
- Make every public fact traceable to official or partner-official evidence.
- Let the generator choose pages from explicit `pageDecision` data.
- Keep official facts, marketplace reference searches, and monetization metadata
  visibly and structurally separate.
- Support thousands or tens of thousands of generated pages without hand-editing
  repeated HTML.

## Package Lifecycle

1. GPT Pro researches one campaign from public sources.
2. GPT Pro writes `research-packages/<campaign-id>.json`.
3. Codex runs `node scripts/validate-research-package.mjs <file>`.
4. Codex imports the package into normalized `data/*.json`.
5. Codex generates the campaign page, selected item pages, metadata, related
   navigation, Marketplace Finder blocks, and sitemap entries.
6. Codex runs site/data/render checks and visual QA.
7. The user and GPT review the pilot before the workflow is used more broadly.

## Top-Level Shape

Required top-level keys:

- `packageVersion`: current value is `0.1`.
- `packageId`: stable campaign-style package id.
- `createdAt`: ISO date when the research package was assembled.
- `createdBy`: tool or model label, such as `gpt-pro-research`.
- `status`: `research-ready-for-codex`, `needs-human-review`, or `draft`.
- `language`: official and summary language metadata.
- `work`: parent work object.
- `campaign`: campaign object.
- `sources`: source records.
- `evidence`: fact-level evidence records.
- `assets`: intended public assets.
- `items`: item group records.
- `pageGenerationPolicy`: generation controls.
- `marketplacePolicy`: marketplace and affiliate constraints.
- `publicVerificationNotes`: public-safe notes that may be rendered on pages.
- `normalizationExpectations`: optional hints about what Codex should preserve,
  derive, or keep review-only during import.
- `unresolvedQuestions`: known uncertainty.
- `humanReviewChecklist`: review checklist for the pilot.

## Evidence Discipline

Every public campaign or item fact needs both a broad `evidenceRefs` entry and
field-level `fieldEvidenceRefs`. Evidence must point to a `Source` whose `type`
is `official` or `partner-official`.

Marketplace sources and secondary search links cannot verify official facts.
They can help users find references after publication, but they must not support
claims about original price, availability, stock, authenticity, store rules,
lineup counts, or campaign periods.

Each evidence record should include:

- `id`
- `sourceId`
- `factType`
- `claimJa` and/or `claimEn`
- `sourceLocator.sectionLabel`
- `sourceLocator.quoteShortJa` or `sourceLocator.note`
- `confidence`
- `reviewRequired`

When GPT Pro is not certain, it must leave the fact out of public fields and add
an `unresolvedQuestions` entry instead.

Field-level evidence is required for public facts such as:

- campaign title, period, start/end dates, partners, categories, and summaries,
- item name, description, lineup/count, original listed price, acquisition
  method, availability label, and start/end dates.

`unresolvedQuestions` is internal review context. Do not write it as public page
copy. Use `publicVerificationNotes` for neutral public caveats.

## Page Decisions

Every item must include:

```json
"pageDecision": {
  "decision": "generate",
  "priority": "high",
  "reason": "Search value and acquisition rule justify a page.",
  "slug": "original-clear-cards",
  "searchValue": "high",
  "factDepth": "high",
  "acquisitionComplexity": "medium",
  "variantExplosionRisk": "low",
  "thinPageRisk": "low"
}
```

Allowed `decision` values:

- `generate`: Codex should generate an item page.
- `parent-card-only`: show the item on the campaign page only.
- `defer`: keep the item out of page generation until missing source, asset, or
  search-value issues are resolved.
- `never`: do not create an item page unless the policy changes.

Use `generate` only when the item has enough unique source-backed facts,
marketplace/search value, or acquisition complexity to avoid a thin page.
Character-variant pages are not generated by default.

`pageGenerationPolicy.itemPages.selectedItemIds`,
`parentCardOnlyItemIds`, and `deferredItemIds` must match item
`pageDecision` values exactly. Codex fails validation when they drift.

## Asset Handling

Assets in a package are plans, not proof of usage rights. Each asset needs:

- `id`
- `role`
- `sourceId`
- `remoteUrl` or `localPath`
- `localPath`
- `altJa`
- `usage`
- `downloadPolicy`
- `rightsNote`

Codex may download, compress, and place images, but it must not publish local
captures, PDFs, browser profiles, or temporary visual-diff files as assets.

## Marketplace Searches

Marketplace links are reference searches. They do not verify facts and must not
claim price, stock, authenticity, or availability.

Each search needs:

- `id`
- `platform`
- `labelJa`
- `labelEn`
- `query`
- `url`
- `intent`
- `finderGroup`
- `region`
- `isAffiliate`
- `rel`
- `disclosureRequired`

Affiliate links are disabled by default. If `isAffiliate` is true, visible
disclosure and `rel="sponsored"` are mandatory.

## Validation Rules

`scripts/validate-research-package.mjs` checks the v0 contract:

- Required top-level objects and arrays exist.
- IDs are unique and lowercase kebab-case.
- `campaign.workId` matches `work.id`.
- source, evidence, asset, and item references resolve.
- public facts use official or partner-official evidence.
- each item has a `pageDecision`.
- each page decision includes `searchValue`, `factDepth`,
  `acquisitionComplexity`, `variantExplosionRisk`, and `thinPageRisk`.
- field-level evidence exists for public campaign and item facts.
- `publicVerificationNotes` exists and does not contain internal Codex/GPT
  instruction wording.
- `publishBlocking: true` unresolved questions block generation.
- `pageGenerationPolicy.itemPages` item id lists match `pageDecision` values.
- generated items have a slug, asset, summary, official source, and search
  metadata.
- page slugs are unique.
- marketplace links keep affiliate and `rel` policy consistent.
- `pageGenerationPolicy` and `marketplacePolicy` keep large-scale generation
  conservative.

The schema file at `schemas/collab-research-package.schema.json` is the
structural contract. The validator enforces the repo-specific rules that JSON
Schema alone does not express well.
